405  <span class="s0">class </span><span class="s1">ExtrasCandidate(Candidate):</span>
406      <span class="s4">&quot;&quot;&quot;A candidate that has 'extras', indicating additional dependencies. 
407   
408      Requirements can be for a project with dependencies, something like 
409      foo[extra].  The extras don't affect the project/version being installed 
410      directly, but indicate that we need additional dependencies. We model that 
411      by having an artificial ExtrasCandidate that wraps the &quot;base&quot; candidate. 
412   
413      The ExtrasCandidate differs from the base in the following ways: 
414   
415      1. It has a unique name, of the form foo[extra]. This causes the resolver 
416         to treat it as a separate node in the dependency graph. 
417      2. When we're getting the candidate's dependencies, 
418         a) We specify that we want the extra dependencies as well. 
419         b) We add a dependency on the base candidate. 
420            See below for why this is needed. 
421      3. We return None for the underlying InstallRequirement, as the base 
422         candidate will provide it, and we don't want to end up with duplicates. 
423   
424      The dependency on the base candidate is needed so that the resolver can't 
425      decide that it should recommend foo[extra1] version 1.0 and foo[extra2] 
426      version 2.0. Having those candidates depend on foo=1.0 and foo=2.0 
427      respectively forces the resolver to recognise that this is a conflict.
