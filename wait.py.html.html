37   <span class="s3"># How should we wait on sockets?</span>
38   <span class="s3">#</span>
39   <span class="s3"># There are two types of APIs you can use for waiting on sockets: the fancy</span>
40   <span class="s3"># modern stateful APIs like epoll/kqueue, and the older stateless APIs like</span>
41   <span class="s3"># select/poll. The stateful APIs are more efficient when you have a lots of</span>
42   <span class="s3"># sockets to keep track of, because you can set them up once and then use them</span>
43   <span class="s3"># lots of times. But we only ever want to wait on a single socket at a time</span>
44   <span class="s3"># and don't want to keep track of state, so the stateless APIs are actually</span>
45   <span class="s3"># more efficient. So we want to use select() or poll().</span>
46   <span class="s3">#</span>
47   <span class="s3"># Now, how do we choose between select() and poll()? On traditional Unixes,</span>
48   <span class="s3"># select() has a strange calling convention that makes it slow, or fail</span>
49   <span class="s3"># altogether, for high-numbered file descriptors. The point of poll() is to fix</span>
50   <span class="s3"># that, so on Unixes, we prefer poll().</span>
51   <span class="s3">#</span>
52   <span class="s3"># On Windows, there is no poll() (or at least Python doesn't provide a wrapper</span>
53   <span class="s3"># for it), but that's OK, because on Windows, select() doesn't have this</span>
54   <span class="s3"># strange calling convention; plain select() works fine.</span>
55   <span class="s3">#</span>
56   <span class="s3"># So: on Windows we use select(), and everywhere else we use poll(). We also</span>
57   <span class="s3"># fall back to select() in case poll() is somehow broken or missing.</span>
58
