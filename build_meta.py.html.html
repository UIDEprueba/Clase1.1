20   <pre><span class="s0">&quot;&quot;&quot;A PEP 517 interface to setuptools 
21    
22   Previously, when a user or a command line tool (let's call it a &quot;frontend&quot;) 
23   needed to make a request of setuptools to take a certain action, for 
24   example, generating a list of installation requirements, the frontend would 
25   would call &quot;setup.py egg_info&quot; or &quot;setup.py bdist_wheel&quot; on the command line. 
26    
27   PEP 517 defines a different method of interfacing with setuptools. Rather 
28   than calling &quot;setup.py&quot; directly, the frontend should: 
29    
30     1. Set the current directory to the directory with a setup.py file 
31     2. Import this module into a safe python interpreter (one in which 
32        setuptools can potentially set global variables or crash hard). 
33     3. Call one of the functions defined in PEP 517. 
34    
35   What each function does is defined in PEP 517. However, here is a &quot;casual&quot; 
36   definition of the functions (this definition should not be relied on for 
37   bug reports or API stability): 
38    
39     - `build_wheel`: build a wheel in the folder and return the basename 
40     - `get_requires_for_build_wheel`: get the `setup_requires` to build 
41     - `prepare_metadata_for_build_wheel`: get the `install_requires` 
42     - `build_sdist`: build an sdist in the folder and return the basename 
43     - `get_requires_for_build_sdist`: get the `setup_requires` to build 
44    
45   Again, this is not a formal definition! Just a &quot;taste&quot; of the module.
